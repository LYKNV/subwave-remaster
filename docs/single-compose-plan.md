# Plan — "Just a docker-compose file" deployment + first-run wizard

## Goal

Make SUB/WAVE deployable from a single `docker-compose.prod.yml` + a tiny `.env`
(three vars), both sitting at the repo root. No clone required; the rest of the
configuration happens in a web wizard on first visit. Operator flow:

```bash
mkdir subwave && cd subwave
curl -O https://raw.githubusercontent.com/perminder-klair/subwave/main/docker-compose.prod.yml
curl -O https://raw.githubusercontent.com/perminder-klair/subwave/main/.env.example
mv .env.example .env   # fill in ADMIN_USER, ADMIN_PASS, SITE_URL — that's it
docker compose -f docker-compose.prod.yml up -d
open https://yourhost.com/                # redirected to /setup wizard
```

The wizard collects Navidrome creds, LLM provider + key, TTS engine, DJ persona,
and offers to generate jingles. Everything it collects lands in
`state/settings.json` — the same file the admin UI already edits. Once the
wizard is done, the operator has a working station.

Dev gets the same `.env`-at-root treatment so there's one config surface across
all three composes (dev, prod, byo-proxy). Dev still uses bind mounts for
hot-reload of `radio.liq` / `Caddyfile`; only prod bakes everything into images.

## Why bundle the wizard with single-compose

Today's "CLI setup" (`scripts/setup.sh`) isn't actually interactive — no `read`
prompts — and the existing `/setup` page is docs-only (`SetupOverview`,
`Prerequisites`, `QuickStart`). The "guided setup" is really just "hand-edit
`controller/.env` with 12 vars in your editor." Shipping single-compose alone
would mean changing install docs twice: first to "edit root `.env` with 12
vars," then later to "actually, configure it in the UI." Bundling them is one
coherent change with one message.

## Decisions locked in

| Question | Choice |
|---|---|
| Config surface (boot) | Single `.env` at repo root with **only** `ADMIN_USER`, `ADMIN_PASS`, `SITE_URL` required. Replaces `docker/.env` + `controller/.env`. |
| Config surface (runtime) | `state/settings.json` is the single source of truth for Navidrome / LLM / TTS / DJ persona / etc. The admin UI already edits this file; the wizard writes to it on first boot. |
| Env vars as overrides | All controller env vars (NAVIDROME_*, ANTHROPIC_API_KEY, …) stay supported as **optional overrides** for 12-factor / CI use cases. If set, they win over `settings.json`. |
| Icecast passwords | Auto-generated by the new icecast image on first boot, written to `state/icecast-secrets.env`, sourced by liquidsoap + controller entrypoints |
| First-run detection | Controller exposes `needsSetup: true` on `/state` when `settings.json` lacks Navidrome creds. Admin shell + landing page redirect to `/setup`. |
| `/setup` page | Rebuilt as an actual wizard (currently a docs index). The existing docs pages move under `/docs/setup/*`. |
| Scope | Both `docker-compose.prod.yml` and `docker-compose.byo-proxy.yml` |
| Dev mode | Also unified onto root `.env`; bind mounts preserved for hot-reload |

## Current dependencies the prod compose has on the repo

`docker/docker-compose.prod.yml` references all of these and breaks without them:

| Dependency | Today | After |
|---|---|---|
| `./Caddyfile` | Bind-mounted from `docker/Caddyfile` | Baked into `ghcr.io/perminder-klair/subwave-caddy` |
| `../liquidsoap/radio.liq` | Bind-mounted from repo | Baked into `subwave-liquidsoap` image. Dev still bind-mounts it. |
| `../sounds/` | Bind-mounted into both liquidsoap + controller | `COPY`'d into both images. Dev still bind-mounts. |
| `${STATE_DIR}/icecast.xml` | Rendered by `scripts/setup.sh` from a template before first boot | New `subwave-icecast` image renders it on boot via `envsubst` entrypoint |
| `../controller/.env` | `env_file:` directive | Replaced by `env_file: ./.env` (root) — lazy passthrough preserved. Runtime config moves to `state/settings.json` via the first-run wizard; env vars become optional overrides. |

## Icecast secrets — coordination details

The icecast image is the canonical source of `ICECAST_*_PASSWORD`. Flow on first boot:

1. `docker compose up -d` starts services in dependency order:
   - `icecast` (no deps) → starts first.
2. Icecast entrypoint:
   ```sh
   SECRETS=/var/sub-wave/icecast-secrets.env
   if [ ! -f "$SECRETS" ]; then
     umask 077
     {
       printf 'ICECAST_SOURCE_PASSWORD=%s\n' "$(openssl rand -hex 16)"
       printf 'ICECAST_ADMIN_PASSWORD=%s\n'  "$(openssl rand -hex 16)"
       printf 'ICECAST_RELAY_PASSWORD=%s\n'  "$(openssl rand -hex 16)"
     } > "$SECRETS"
   fi
   . "$SECRETS"
   envsubst < /etc/icecast2/icecast.xml.template > /etc/icecast2/icecast.xml
   exec icecast2 -c /etc/icecast2/icecast.xml
   ```
3. `liquidsoap` waits on `depends_on: [icecast]`. Its entrypoint blocks until `/var/sub-wave/icecast-secrets.env` exists, sources it, then execs liquidsoap. (Short bounded wait — 30s, then bail loudly.)
4. `controller` waits on `depends_on: [liquidsoap]`; its entrypoint does the same source + exec.

**Operator override:** if the root `.env` declares `ICECAST_SOURCE_PASSWORD` etc., compose's `environment:` block passes them into the icecast container. The icecast entrypoint then **uses the env-provided values** instead of generating new ones. This means a user who really wants to set them in `.env` can; the magic only kicks in when they're blank.

**Reset path:** to rotate Icecast passwords, delete `state/icecast-secrets.env` and restart **all three of icecast, liquidsoap, and controller** — the latter two cache the env at process start, so an icecast-only restart leaves them on the old secrets. Documented in CLAUDE.md.

## First-run wizard — flow

Triggered by the controller exposing `needsSetup: true` on `/state` when
`state/settings.json` is missing Navidrome credentials. Operator flow:

1. **Land on `/`** — `useStationFeed` polls `/state`, sees `needsSetup: true`, the landing page renders a banner ("Welcome to SUB/WAVE — finish setup →"). Clicking takes them to `/setup`.
2. **`/setup`** — checks `needsSetup` itself. If false, renders `<SetupOverview />` (the existing docs index). If true, renders `<Wizard />`.
3. **Admin login** — `useAdminAuth` prompts for `ADMIN_USER`/`ADMIN_PASS` (the values from `.env`). Stored in `localStorage` per existing pattern.
4. **Step 1 — Navidrome.** Form fields: URL, username, password. "Test connection" hits `POST /setup/test-navidrome`. Green check → "Next" enabled. Skip allowed (the wizard tags the setup as "music source pending" and the controller falls back to local files — power user path).
5. **Step 2 — LLM.** Provider picker (Ollama / OpenAI-compatible / Anthropic / OpenAI / Google / DeepSeek / OpenRouter / Gateway), model name, optional API key. Ollama is the default and needs no key. "Test" hits `POST /setup/test-llm`.
6. **Step 3 — TTS.** Engine picker (Piper / Kokoro / Cloud / Chatterbox). Cloud asks for an OpenAI or ElevenLabs key. Chatterbox is only offered if the image was built with `WITH_CHATTERBOX=1` (controller advertises this on `/state`).
7. **Step 4 — DJ persona.** Station name (e.g. "SUB/WAVE"), location ("London, UK"), and a soul textarea pre-filled with one of the seeded `DJ_SOULS`. Frequency picker (quiet / moderate / aggressive).
8. **Step 5 — Jingles.** One-click "Generate jingles now" button → `POST /setup/generate-jingles`. Shows progress (the same render flow the admin UI already drives). Skippable — operator can do it later from `/admin/jingles`.
9. **Review + save.** `POST /setup/save` with the full collected object. Controller persists to `settings.json`, sets `setupCompletedAt`, flips `needsSetup` to false. Wizard redirects to `/admin` with a "Setup complete — your station is live at /listen" toast.

All wizard endpoints are behind `requireAdmin`. The wizard cannot be reached
without `ADMIN_USER` / `ADMIN_PASS` being set in `.env` — which is enforced at
controller startup in prod mode anyway, so this is consistent with the existing
security posture.

## Migration for existing operators

Anyone already running with `docker/.env` + `controller/.env`:

1. **`scripts/setup.sh`** detects the old layout and merges the three required vars (`ADMIN_USER`, `ADMIN_PASS`, `SITE_URL`) plus any cloud LLM/TTS keys into a root `.env` (idempotent, won't overwrite if root `.env` already exists). Anything else (Navidrome creds, TTS speed, news feed URL, …) is ignored — the controller's first-boot migration handles those.
2. **Controller first-boot migration** (`controller/src/setup/migrate.js`, new): if `settings.json` is missing fields that have a matching env var, copy them in once and set `setupCompletedAt`. Net effect: an upgrading operator with Navidrome creds in `controller/.env` boots with a populated `settings.json` and `needsSetup: false` — no wizard, nothing to do.
3. The old `docker/.env` and `controller/.env` files are NOT auto-deleted — the migration is one-way, but the old files become unused. Document this in the release notes.
4. `state/icecast.xml` becomes unused but is harmless; can be deleted manually.
5. CLAUDE.md gets a migration note covering both the compose-path change and the config-surface change.

## Tradeoffs accepted

1. **Editing `radio.liq` in prod now requires an image rebuild**, not a `docker compose restart liquidsoap`. Dev compose keeps the bind mount, so day-to-day Liquidsoap work is unaffected. Acceptable because prod tracks tagged releases (`SUBWAVE_VERSION`) anyway.
2. **Editing `Caddyfile` in prod requires an image rebuild.** Same reasoning. Caddy config changes are rare.
3. **5 images to publish instead of 3.** Caddy + Icecast images are tiny and rarely change; cache will hit on most builds.
4. **`docker compose` invocation moves to repo root**, breaking muscle memory for `cd docker && docker compose ...`. README + CLAUDE.md updated; no functional impact.
5. **`web/.env.local` stays separate** because Next.js native `npm run dev` reads from `web/` and we don't want to teach it about a root `.env`. Documented.
6. **Two sources of truth (env vs settings.json) for the same fields.** Env wins when set. Documented precedence everywhere it matters; the wizard form labels say "(override in `.env` will take precedence)" next to a field if the controller detects an override at form-load time. Slightly more code than env-only or settings-only, but the alternative (force everyone onto one or the other) breaks one of the two install paths.
7. **Cloud LLM/TTS API keys collected via web form get written to `state/secrets.env`.** Plain text on disk, mode 0600. Same risk profile as `state/icecast-secrets.env`. An operator who wants stronger key handling can set env vars instead and skip the wizard fields — `state/secrets.env` only gets written if the wizard form supplied a value the env didn't already have.
8. **Wizard requires admin auth to reach.** This means the operator has to put `ADMIN_USER` + `ADMIN_PASS` in `.env` before they can even start the wizard. Already mandatory in prod, but worth calling out for first-time operators — the error message has to be very clear if `ADMIN_PASS` is missing.

## Out of scope

- Multi-arch (arm64) images — Piper/Kokoro/Chatterbox wheels are amd64-only; same constraint as today.
- A `curl | sh` installer — easy to add later once the two-file install works.
- Renaming any of the three first-party images.
